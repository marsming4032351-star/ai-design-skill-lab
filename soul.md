# soul.md：AI Design Skill Lab 的灵魂文件

这个仓库不是一个普通代码仓库。

它是一个 **AI Design Skill Lab / Design Data Factory**：把设计经验、AI 工作流、模板、判断标准、运行记录和发布成果沉淀为可复用资产的实验室。

这里的目标不是把更多素材堆进来，也不是让 AI 随机生成更多图片。这里真正要积累的是：设计工作如何被理解、拆解、执行、评估、复用和版本化。

## 核心定位

这个仓库不是素材库。

素材库只回答“有什么图、有什么文件”。这个仓库要回答“这些素材为什么可用、适合什么场景、如何被再次调用、如何判断它是否值得复用”。

这个仓库不是单纯生图工具。

单次生图只能产生结果。这个仓库更关心结果背后的流程：目标是什么、参考从哪里来、Prompt 如何形成、HTML/CSS 如何承接排版、Playwright 如何验证导出、最终 PNG 是否可发布。

这个仓库不是普通网页模板仓库。

模板只有在被理解之后才有价值。外部模板要经过分析、评估、隔离和选择性吸收，才能进入本项目的工作流。`references/beautiful-html-templates` 的评估就是这个原则的例子：它可以作为 PPT / HTML deck 的原比例模板参考，也可以被转译成小红书 4:5 卡片，但不能不加判断地直接照搬。

这个仓库是一个设计生产实验室。

它把设计经验拆成结构化资产：规则、Prompt、模板、案例、运行记录、评估文档、图文成品和可验证的导出流程。它让 Codex、Claude Code 和人类合作者可以在同一个知识空间里工作，而不是每次从零开始。

## 核心信念

### 设计不只是结果，而是可复用的数据流

设计不是最后那张图，也不是一次灵感爆发。设计是一条数据流：

```text
brief -> reference -> analysis -> plan -> prompt -> layout -> render -> review -> archive -> reuse
```

Design Data Factory 的价值在于把这条链路显性化。一个优秀结果如果不能被复盘、比较、重用，它就只是一次性输出；一个普通结果如果带着清晰的判断过程，也可能成为下次工作的起点。

### AI 不只是生成工具，而是参与工作流的执行角色

Codex 和 Claude Code 不应该只被当作“帮我写代码”或“帮我生成图片”的按钮。它们是执行型 Agent：

- 读仓库；
- 分析外部项目；
- 拆解模板机制；
- 编写 HTML/CSS；
- 调用 Playwright 导出 PNG；
- 检查 `git status`；
- 整理评估文档；
- 把工作结果沉淀回 docs 和 references。

AI 的价值不是替代判断，而是把判断之后的执行链路变得更稳定、更可审计。

### Git 不只是代码管理，而是设计资产的版本系统

Git 在这个仓库里记录的不只是代码 diff。它记录的是设计资产如何演化：

- 一套小红书图文如何从主题变成 HTML，再导出 PNG；
- 一个开源项目如何从外部链接变成本地 reference 和评估文档；
- 一条设计规则如何影响后续模板选择；
- 一次 Prompt、一次评估、一次导出如何成为未来复用的依据。

`git commit` 是本地资产里程碑，`git push` 是外部发布边界。AI 可以协助 commit，但不应该默认 push。越是 AI 加速的工作流，越需要清晰的 Git 边界。

### 外部开源项目不能直接照搬

看到一个 GitHub 开源项目，第一步不是复制。正确顺序是：

```text
先分析 -> 再克隆 -> 放入 references -> 阅读 README / AGENTS / index -> 写评估 -> 决定是否吸收
```

完整第三方仓库通常不应该直接作为本项目源码提交。它可以作为本地 reference，可以写入 `.gitignore`，可以在文档里保留来源、用途和判断结论。真正应该沉淀的是本项目自己的评估、选择机制和接入方式。

## 文件职责

### `README.md`

`README.md` 是项目的入口说明。它告诉后来者这个仓库当前是什么、能运行什么、核心架构是什么、如何快速开始。

它偏向“操作入口”和“当前状态”，应该保持可执行、可导航、不过度哲学化。

### `AGENTS.md`

如果存在 `AGENTS.md`，它应该是 Agent 的工作协议：Codex、Claude Code 或其它执行型 Agent 进入仓库后，应该遵守哪些边界、检查哪些文件、如何修改、如何验证、如何提交。

它应该写“怎么做”，而不是写“为什么存在”。`soul.md` 负责为什么，`AGENTS.md` 负责怎么协作。

### `soul.md`

`soul.md` 是项目的价值观和判断标准。

它回答：

- 这个仓库为什么存在；
- 什么值得沉淀；
- 什么不应该进入仓库；
- 人和 Agent 如何分工；
- 什么样的外部项目可以被吸收；
- 为什么 Git、docs、references 和导出成果都属于设计资产系统的一部分。

### `docs/`

`docs/` 是知识资产沉淀空间。

这里保存：

- 架构说明；
- Harness Runtime 文档；
- 小红书图文包；
- 开源项目评估；
- Feishu / Obsidian / 工作流说明；
- HTML/CSS 到 PNG 的可发布成果；
- 设计案例、内容逻辑和复盘文档。

`docs/` 不只是说明书目录，它是项目的知识库。对外可读、对内可复用的判断，都应该优先沉淀在这里。

### `skills/`

`skills/` 应该存放可复用的 Agent 能力说明、操作流程和专门技能。

一个 skill 的价值不在于“写了一段提示词”，而在于它能让 Agent 在类似任务里稳定重复一套高质量行为：如何分析、如何生成、如何验证、如何交付。

### `templates/`

`templates/` 应该存放项目自己认可、整理过、可直接复用或改造的模板。

外部模板不能因为好看就直接进入 `templates/`。它必须先经过 `references/` 隔离、文档评估、适用场景判断和必要的改造规范。只有成为项目内生资产后，才适合放入 `templates/`。

### `references/`

`references/` 是外部参考和内部样本的隔离区。

这里可以存放：

- 示例项目；
- 规则；
- Prompt；
- Patterns；
- schemas；
- run 记录；
- 外部开源项目 checkout。

`references/` 的核心原则是隔离。它允许我们研究外部项目，但不默认把它们变成本项目资产。外部仓库要先被理解，再被选择性吸收。

### `runs/`

`runs/` 如果存在，应该保存运行产物、过程记录和可复盘的执行历史。

运行记录的价值不在于“留痕”，而在于以后能回答：

- 这次目标是什么；
- Agent 做了什么；
- 哪些输入被消耗；
- 哪些判断导致了结果；
- 哪些输出值得进入 archive / pattern / docs。

如果运行产物只是临时缓存、不可复盘截图或无解释中间文件，就不应该进入长期沉淀区。

## 什么值得沉淀

值得沉淀的内容，必须能提升未来工作的判断力或执行稳定性。

包括：

- 方法论：能够解释一类设计问题如何被拆解和处理。
- 可复用模板：经过验证，能在多个场景中重复使用的 HTML、PPT、卡片、提案或图文结构。
- 可复用 Prompt：不是一次性句子，而是带有场景、输入、输出和质量边界的 Prompt。
- 评估文档：对外部项目、模板、工具、工作流的分析结论。
- 工作流案例：从目标、参考、执行到导出的完整过程。
- 设计判断标准：什么算适合，什么算不适合，什么情况下应该拒绝接入。
- 验证方法：例如 HTML/CSS -> Playwright -> PNG 的导出检查、尺寸检查、文字溢出检查。

沉淀的标准不是“有没有用过”，而是“以后是否能减少重复判断、提升执行质量”。

## 什么不应该沉淀

不应该进入长期资产的内容包括：

- 临时文件、缓存、`.DS_Store`、`__pycache__`、Playwright 临时输出。
- 未验证的第三方源码。
- 一次性截图草稿。
- 无来源、无用途、无判断过程的素材堆积。
- 只为当前聊天存在、没有复用价值的中间文件。
- 带有 token、secret、auth cache、本机绝对路径或私人工作区信息的内容。
- 没有 README、没有使用说明、没有边界说明的“看起来不错”的模板。

如果一个内容不能说明“为什么值得留下”，它就不应该默认进入仓库。

## 人和 Agent 的分工

### 人负责方向、审美、商业判断和最终选择

人类合作者负责定义目标、判断品味、理解商业语境、决定是否发布、是否接入、是否提交。

AI 可以提出选项，但不能替代最终判断。尤其是品牌气质、内容取舍、审美方向、客户语境和公开发布边界，必须由人确认。

### Codex 负责分析、执行、整理、验证和导出

Codex 是这个仓库里的主要执行型 Agent。

它适合承担：

- 读取仓库和文档；
- 分析 GitHub 外部项目；
- 克隆到 `references/`；
- 写评估文档；
- 生成 HTML/CSS 图文；
- 调用 Playwright 导出 PNG；
- 检查文件、尺寸、git status；
- 整理提交范围；
- 把一次工作变成可复用资产。

Codex 应该主动保护边界：不随手 `git add .`，不提交完整第三方仓库，不把临时文件当资产。

### Claude Code 可负责复杂重构和长文档规划

Claude Code 适合承担更长链路的规划、复杂重构、跨文件文档结构设计和大型内容组织。

当任务需要长期上下文、成体系改造或多阶段文档规划时，Claude Code 可以作为更偏架构和长文档整理的协作者。

### Git 负责记录资产演化

Git 是这个实验室的资产时间线。

每次 commit 都应该尽量表达一个清楚的资产变化：新增一套图文包、完成一次模板评估、沉淀一个工作流、更新一个设计规则，而不是把无关文件混在一起。

Git 历史越干净，这个仓库越像设计资产系统；Git 历史越混乱，它就越退化成文件堆。

## 小红书图文工作流的意义

小红书图文不是“顺手做内容”。它是这个仓库验证设计生产链路的高频场景。

当前稳定路径是：

```text
主题判断 -> 内容结构 -> GPT Image 氛围背景 -> HTML/CSS 中文排版 -> Playwright 截图 -> 1080 x 1350 PNG -> post-copy.md
```

这个路径的重点不是图片数量，而是把内容、视觉、排版和发布文案变成一组可修改、可复查、可再次导出的资产。

HTML/CSS 负责稳定排版，Playwright 负责真实浏览器导出，PNG 负责发布交付。这样比直接在生图里写中文更可靠，也更适合长期迭代。

## Obsidian、docs 和知识空间

Obsidian 和 `docs/` 是这个项目的知识资产层。

不是所有内容都应该进入代码路径，但有判断价值的内容应该进入知识空间：项目复盘、工作流说明、模板评估、方法论图文、外部项目分析、AI 使用指南。

知识资产的目标不是“记录很多”，而是让未来的 Agent 和人类合作者更快进入上下文，少重复踩坑。

## 项目原则

### 先分析，再接入

任何外部项目、模板、工具、Prompt，都先判断它解决什么问题、适合什么场景、有什么边界，再决定是否进入工作流。

### 先隔离，再吸收

外部仓库先进 `references/`。完整 checkout 不直接变成本项目源码。经过评估后，只吸收机制、模板语法、判断标准或必要的改造结果。

### 先沉淀方法，再追求数量

多生成几张图、多复制几个模板，都不是核心进展。真正的进展是沉淀一套以后还能复用的判断方法和执行路径。

### 先形成资产，再形成自动化

自动化只能放大已有系统。没有清晰资产结构、文件职责和验证标准之前，不急着做自动化。先把一次高质量工作做清楚，再把它变成脚本、skill、模板和运行时能力。

### 让每次输出都能回到系统

一张图、一篇小红书、一份评估、一次 Git 提交，都应该尽量回到同一个资产系统里：可读、可查、可复用、可验证、可演化。
