01 · 这个仓库在做什么 purpose

它不是素材库，
而是一个 AI 设计工作台。

仓库的目标很简单：把一次设计工作里散落的资料、Prompt、规则、运行记录和输出成果，沉淀成未来可以复用的资产。

Design Data Factory

把原始资料加工成结构化资产，像工厂一样一层层沉淀。

Harness Runtime

模拟一条任务怎么跑完，让 Agent 按生命周期做事。

Output Skill

把输出整理成可发布、可演示、可归档的成品包。

MAP.md

告诉你仓库入口在哪，先读什么，再走哪条路。

一句话：它想解决的不是“会不会生成”，而是“生成完能不能留下来”。

02 · 仓库的四层结构 structure

先分层，
再理解每个目录的职责。

聊天里讲的“资料层、资产层、生产层、任务与输出层”，其实是在说：不同文件有不同任务，不要混在一起。

资料层 → 先收集

`00_Inbox_Staging`、`imports`、`references`：先把输入放进来，再判断值不值得吸收。

资产层 → 再沉淀

`docs`、`templates`、`prompts`、`rules`：把方法、模板、规范和说明变成可复用资产。

生产层 → 负责执行

`scripts`、`harness`、`tests`：把事情跑起来、验起来，而不是只写说明。

任务与输出层 → 负责路由和归档

`MAP.md`、`AGENTS.md`、`CLAUDE.md`、`soul.md`、`CHANGES`：先指路，再记录价值观和历史。

简单理解：先接住资料，再加工成资产，最后通过流程输出成品。

03 · 输出技能怎么分工 output skill

Output Skill 不是模板，
而是 输出生产流水线。

它关心的是：怎么把内容先规划，再设计，再构建，再验证，最后导出成能交付的东西。

Plan → Design → Build → Validate → Export
它能管什么

HTML 演示页、PPT 风格图文、小红书图文、PDF、视频脚本文档。

它不干什么

它不替代主流程代码，也不复制第三方仓库的实现。

这就是为什么它和 `docs/OUTPUT_SKILL_SPEC.md` 是一类东西，而不是一张成品图。

04 · 生图链路怎么分工 image vs lovart

你问的生图工具，
其实有 两条线。

`imagegen` 是系统里的通用生图能力；`design-generate` 才是当前仓库里接 Lovart API 的那条视觉生产线。

imagegen

偏 OpenAI 图像生成能力，用来直接出图或改图。

design-generate

仓库里的视觉生产流程，对接 `shared/lovart_hook.py` 和 `scripts/generate_design.py`。

Lovart

真实的生图后端，适合 live 模式；没有 key 时会走 mock 占位图。

你该怎么说

要 Lovart：说“用 design-generate 走 live 模式”。要占位图：说“走 mock 模式”。

最后一句：先搞清楚工具分工，再决定是出图、归档，还是把方法沉淀成规范。
