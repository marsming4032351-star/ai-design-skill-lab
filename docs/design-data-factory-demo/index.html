01 · WHY 问题 / The Pain

AI 让"生成"几乎免费,
却让"沉淀"几乎不可能。

真正的代价不在算力,而在每一次生成之后——上下文、判断和经验都消失了。

01 上下文散落 Prompt、参考图、聊天记录散在不同工具里,事后无法还原"为什么这样做"。
02 判断不可见 采纳与否决的理由停留在脑中和私聊里,无法被复查、被引用、被传承。
03 经验不回流 每个项目从零开始问、找、试,上一次的最佳实践无法成为下一次的起点。
所以问题不是"AI 不会做",而是"做完之后什么都不剩下"。

05 · HARNESS 任务生命周期 / State Machine

Harness 是任务的外壳,
负责"任务怎么跑完"。

一个 deterministic 状态机:把 AI 设计任务从目标到归档,变成五个可观测节点。

STATE 01
Goal
明确本次任务要解决的问题与约束。
→ planner
STATE 02
Planner
拆解目标,给出可执行的子步骤。
→ generator
STATE 03
Generator
生成候选设计或 mock 结果。
→ critic
STATE 04
Critic
评价结果;不达标则回到 generator 重试。
↺ retry / ↓
STATE 05
Review
人工边界停点;通过后归档进资产库。
→ archive
当前是 deterministic mock lifecycle——真实任务流转,但不调用 Pipeline CLI。
关注点:状态流转 · 失败重试 · 人工边界 · 任务可观测 06 / 12

08 · END TO END 闭环 / The Loop

一次任务,从 brief 进来,
到 pattern 出去——闭合。

关心的不是"这次做完",而是"做完之后,下一次能不能更快、更稳、更准"。

接收
目标、资料、限制、场景进入 staging,先隔离不动。
→ 00_Inbox_Staging
REF · 参考
判断
外部资料是否值得吸收,变成可复用资产。
→ references/
RUN · 计划
生成
产出方向 Plan,记录消耗了哪些资产与决策。
→ harness / plan
OUT · 渲染
交付
转换为 HTML、图文、deck 或 visual 输出包。
→ render / deliver
LOOP · 沉淀
回流
通过评价的判断,沉淀为 Pattern / Rule。
↺ pattern lib
BRIEF → REFERENCE → PLAN → RENDER → REVIEW → ARCHIVE ↺ REUSE — 下一次的起点
每一次任务,都在让下一次更便宜。 09 / 12
09 · MAP 仓库导航 / How To Read

从一句话和一份地图开始。

不必读完所有文件——MAP.md 是第一事实源,其他目录围绕资料、资产、生产、输出各司其职。

DATA · 资产 references/ Pattern、Rule、Prompt、schema、Run 历史记录。
CLI · 工具 scripts/ Pipeline 单步 CLI:scan / run / critic / archive / generate。
CORE · 共享 shared/ schema 解析、prompt render、rule / rubric engine 等共用能力。
RUNTIME · 运行 harness/ Runtime 节点:planner / generator / critic / review。
给新人一句话:先看 MAP,再按任务走目录。 10 / 12
